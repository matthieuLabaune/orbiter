## Audit Date: 2026-03-25
## Auditor: Claude Code (Playwright navigation)

## BUGS CRITIQUES

### BUG-1: Page Tests & V&V — contenu vide, erreurs Alpine
- **Page**: `/projects/{slug}/tests`
- **Symptôme**: La liste des tests et la matrice de traçabilité ne s'affichent pas. Seuls les onglets "Liste" / "Matrice de traçabilité" sont visibles.
- **Erreur console**: `ReferenceError: tab is not defined` — le `x-data` Alpine avec `tab` est sur un parent qui ne wrape pas les onglets correctement. Le `x-show="tab === 'list'"` fait référence à `tab` mais il est dans un scope Alpine différent.
- **Cause**: Le `x-data="{ tab: '...' }"` a été ajouté dans la vue `tests/index.blade.php` mais le contenu existant de la liste (table Blade) n'est pas wrappé dans le `x-show` div correctement, et les directives `x-cloak` masquent le contenu.
- **Priorité**: P0 — une page entière est cassée

### BUG-2: Warning Alpine "Detected multiple instances"
- **Console**: `Detected multiple instances of Alpine running` sur TOUTES les pages
- **Cause**: Alpine.js est chargé deux fois — une fois via `app.js` (import Alpine from 'alpinejs') et une fois via Livewire (qui embarque Alpine)
- **Priorité**: P1 — peut causer des comportements erratiques

### BUG-3: Login/Register pages non stylées dark mode
- **Page**: `/login`, `/register`
- **Symptôme**: Pages en fond gris clair (#f3f4f6), le logo Orbiter est gris/invisible, pas de dark mode
- **Cause**: Les pages guest de Breeze n'ont pas été mises à jour avec les classes `dark:`
- **Priorité**: P1 — première impression de l'app

## DESIGN & UX — PROBLÈMES

### DESIGN-1: Light mode incomplet
- **Pages affectées**: Toutes
- **Problèmes**:
  - La zone Mermaid auto-générée garde un fond dark en mode clair
  - Les cartes projet sur l'index gardent des bordures dark
  - La top nav (Breeze) n'a pas de variantes dark/light
  - Les formulaires (create/edit) gardent des inputs dark (bg-slate-800)
  - Les badges V&V, priority, risk score n'ont pas de variantes light
  - Le header de page n'a pas de titre visible en mode light sur certaines pages
- **Impact**: Le mode clair est partiellement cassé — visuellement incohérent

### DESIGN-2: Navigation top redondante avec sidebar
- Le header de Breeze (avec "Dashboard" et "Demo User") est toujours visible en plus de la sidebar
- Redondance : le Dashboard link est dans la sidebar ET dans le header
- Le dropdown "Demo User" dans le header n'a pas de fonctionnalité visible (pas de logout, settings)
- **Recommandation**: Retirer le header Breeze quand la sidebar est présente, ou l'intégrer dans la sidebar

### DESIGN-3: Sidebar — pas de sélecteur de projet fonctionnel
- Le bloc "Projet: Orbiter v1" en haut de la sidebar est passif
- On ne peut pas changer de projet depuis la sidebar sans revenir au dashboard
- **Recommandation**: Dropdown de sélection de projet dans la sidebar

### DESIGN-4: Dashboard projet — densité d'information
- Le dashboard est riche mais manque de hiérarchie visuelle
- Les cartes module health sont toutes de la même taille même si elles ont des données très différentes
- Le widget "Activité récente" montre trop de `27 minutes ago` (timestamps du seeder)
- Manque un indicateur visuel "résumé" en haut (ex: jauge globale d'avancement)
- **Recommandation**: Ajouter une jauge circulaire d'avancement global, condenser les modules health

### DESIGN-5: Tables — manque de pagination
- Les tables (requirements, tests, anomalies, tasks) n'ont pas de pagination
- Avec 16 requirements c'est OK, mais avec 100+ ce sera inutilisable
- **Recommandation**: Ajouter pagination Livewire (10/25/50 par page)

## UX — AMÉLIORATIONS

### UX-1: Recherche globale manquante
- Aucune barre de recherche globale pour trouver une REQ, un test, une tâche par ref ou titre
- L'utilisateur doit naviguer dans chaque section pour trouver quelque chose
- **Recommandation**: Cmd+K / barre de recherche globale dans le header

### UX-2: Breadcrumbs manquants
- La navigation ne montre pas le chemin (Orbiter v1 > Requirements > REQ-003)
- Le bouton retour (flèche) est présent mais pas toujours cohérent
- **Recommandation**: Breadcrumbs dans le header de chaque page

### UX-3: Formulaires — pas de feedback en temps réel
- Les formulaires de création (requirement, test, task) n'ont pas de validation côté client
- Pas de compteur de caractères, pas de preview markdown
- Les selects de module/requirement pourraient avoir un search
- **Recommandation**: Validation Livewire temps réel, searchable selects

### UX-4: Module show — les liens REQ ne fonctionnent pas
- La page modules/show a des liens `#` pour les requirements au lieu de vraies URLs
- **Priorité**: P2 — liens cassés

### UX-5: Gantt chart — pas testé visuellement
- L'onglet Gantt dans Planning utilise frappe-gantt via @script de Livewire
- Le import dynamique de frappe-gantt peut échouer selon la config Vite
- **Recommandation**: Tester, fallback gracieux si le JS ne charge pas

### UX-6: Pas de notifications/toasts
- Les messages de succès (`session('success')`) ne sont pas visibles après un redirect Livewire
- **Recommandation**: Toast notification component

## LANDING PAGE — AMÉLIORATIONS

### LP-1: Manque de cas d'usage concrets
- La landing montre les concepts mais pas d'exemples visuels de l'app
- Pas de screenshots, pas de démo interactive
- **Recommandation**: Section "See it in action" avec des screenshots annotés

### LP-2: Pas de comparaison Orbiter vs gestion classique
- Le différenciateur n'est pas clair pour un visiteur non familier avec l'ingénierie système
- **Recommandation**: Tableau comparatif "Gestion classique (Jira/Linear) vs Orbiter"

### LP-3: Pas de documentation process
- Le "Comment ça marche" est trop simplifié (4 étapes)
- Manque une page /docs ou /methodology accessible publiquement
- **Recommandation**: Page dédiée expliquant le process avec diagrammes

### LP-4: Le logo SVG est trop discret
- Le planet logo est petit et peu visible dans la nav
- La landing ne met pas le logo en valeur

## TECHNIQUE

### TECH-1: Double Alpine.js
- Retirer l'import Alpine de `app.js` — Livewire l'inclut déjà

### TECH-2: Tailwind CSS 4 vs config
- Le `tailwind.config.js` utilise la syntaxe Tailwind 3 (`@tailwind base` etc)
- Tailwind 4 utilise `@import "tailwindcss"` — vérifier la compatibilité

### TECH-3: Pas de .env.testing
- Les tests nécessitent des env vars manuelles pour fonctionner dans Docker
- **Recommandation**: Créer `.env.testing` avec SQLite config
